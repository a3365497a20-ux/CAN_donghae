# 개발 방법론

**프로젝트명:** ICT 교내 자원 내비게이션 시스템 (CampusNav)
**기관:** ICT폴리텍대학 AI소프트웨어학과
**팀원:** 박승순 · 권동해 · 원태연 · 이수혁
**작성일:** 2026-05-27
**버전:** v1.0

---

## 1. 개발 방법론 개요

### 1.1 채택 방법론

본 프로젝트는 **애자일(Agile) 기반의 반복 점진적 개발 방법론**을 채택한다.

소규모 팀(4인), 단기 개발 일정, 요구사항 변경 가능성을 고려하여 기능 단위로 반복(Iteration)하며 점진적으로 완성도를 높이는 방식을 적용한다. 전통적인 폭포수(Waterfall) 모델의 단계별 산출물 체계는 유지하되, 구현과 피드백을 병행하는 유연한 방식으로 진행한다.

### 1.2 방법론 선택 근거

| 고려 요소 | 내용 | 애자일 적합 이유 |
|-----------|------|-----------------|
| 팀 규모 | 4인 소규모 팀 | 의사소통 오버헤드 최소화 |
| 개발 기간 | 단기 집중 개발 | 빠른 반복 주기로 완성도 향상 |
| 요구사항 | 초기 요구사항 변경 가능성 존재 | 변경에 유연하게 대응 가능 |
| 기술 숙련도 | JSP/Servlet 환경 경험 필요 | 구현하며 학습 병행 가능 |
| 피드백 | 사용자 역할별 UI/UX 검증 필요 | 기능 단위 시연 및 즉각 피드백 가능 |

## 2. 개발 프로세스

### 2.1 전체 개발 단계

```
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│  1단계   │ → │  2단계   │ → │  3단계   │ → │  4단계   │ → │  5단계   │
│  분석    │   │  설계    │   │  구현    │   │  테스트  │   │  배포    │
│ Analysis │   │  Design  │   │  Coding  │   │  Testing │   │ Deploy   │
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
```

각 단계는 순차적으로 진행되며, 구현~테스트 구간은 기능 단위로 반복 수행한다.

---

### 2.2 단계별 상세 내용

#### 1단계 — 분석 (Analysis)

**목적:** 시스템이 해결해야 할 문제와 요구사항을 명확히 정의한다.

| 활동 | 내용 | 산출물 |
|------|------|--------|
| 현황 조사 | 교내 자산 현황, 사용자 역할, 기존 관리 방식 파악 | 현황분석서 |
| 환경 분석 | 기술 스택, DB 구조, 서버 환경 분석 | 환경분석서 |
| 요구사항 도출 | 기능 요구사항(FR), 비기능 요구사항(NFR) 정의 | 요구사항 명세서 |
| 제약사항 확인 | 기술 스택 고정, 외부 API 미연동 등 제약 조건 확인 | 요구사항 명세서 |

#### 2단계 — 설계 (Design)

**목적:** 분석 결과를 바탕으로 시스템 구조와 DB 스키마를 설계한다.

| 활동 | 내용 | 산출물 |
|------|------|--------|
| 시스템 아키텍처 설계 | JSP(View) + Servlet(Controller) + MySQL(Model) 계층 구조 설계 | 아키텍처 다이어그램 |
| DB 설계 | 테이블 정의, 컬럼 설계, 외래키 관계 설정 (10개 테이블) | ERD, DDL SQL |
| UI/UX 설계 | 역할별 화면 흐름, 공통 디자인 시스템(ppd4) 정의 | 화면 설계서 |
| URL 설계 | 서블릿 URL 매핑, JSP 페이지 연결 구조 설계 | web.xml 설계 |
| 보안 설계 | 세션 기반 인증, 역할별 접근 제어 방식 설계 | 보안 설계서 |

#### 3단계 — 구현 (Implementation)

**목적:** 설계를 바탕으로 실제 동작하는 코드를 작성한다.

구현은 **기능 모듈 단위**로 순서를 정해 진행하며, 각 모듈은 독립적으로 동작 가능하도록 구성한다.

| 구현 순서 | 모듈 | 주요 파일 |
|-----------|------|-----------|
| 1 | DB 구축 | campusnav_db.sql → assets.sql → transfer.sql → add.sql |
| 2 | 공통 유틸 | DBUtil.java, web.xml, common.css |
| 3 | 인증 모듈 | LoginServlet.java, LogoutServlet.java, campuslogin.jsp, register.jsp |
| 4 | 역할별 메인 | main_student.jsp, main_assistant.jsp, main_professor.jsp, main_admin.jsp |
| 5 | 자산 검색 | SearchServlet.java, search.jsp, DetailServlet.java, detail.jsp |
| 6 | 예약 모듈 | ReserveServlet.java, reserve.jsp, room_reserve.jsp |
| 7 | 이관 조회 | transfer.jsp |
| 8 | 교수 자원 | professor.jsp |
| 9 | 관리자 기능 | asset_manage.jsp, disposal_admin.jsp, reservations_admin.jsp |
| 10 | 길찾기 | navigationTest1.jsp, floorNav.jsp, saveRoute.jsp, saveFloorRoute.jsp |
| 11 | 게스트·방문자 | GuestServlet.java, VisitorServlet.java, main_guest.jsp, main_visitor.jsp |

#### 4단계 — 테스트 (Testing)

**목적:** 구현된 기능이 요구사항을 충족하는지 검증한다.

| 테스트 유형 | 방법 | 대상 |
|-------------|------|------|
| 단위 테스트 | 모듈별 단독 기능 동작 확인 | 각 Servlet, JSP 페이지 |
| 통합 테스트 | 로그인 → 검색 → 예약 전체 흐름 확인 | 역할별 시나리오 |
| 역할별 테스트 | 6개 역할 테스트 계정으로 접근 권한 확인 | 인증·권한 모듈 |
| DB 연동 테스트 | 실제 DB 데이터 조회·저장·수정 확인 | CRUD 전체 기능 |
| UI 테스트 | PC·모바일 브라우저 레이아웃 확인 | 반응형 화면 |
| 예외 처리 테스트 | 중복 예약, 잘못된 입력, 세션 만료 상황 테스트 | 예약·인증 모듈 |

#### 5단계 — 배포 (Deployment)

**목적:** 완성된 시스템을 실제 운영 환경에 설치하고 운영 가능 상태로 만든다.

| 활동 | 내용 |
|------|------|
| SQL 파일 순서 실행 | campusnav_db.sql → assets.sql → transfer.sql → add.sql 순서대로 실행 |
| Tomcat 배포 | CampusNav 폴더를 webapps/ 하위에 복사 |
| Servlet 컴파일 | compile.bat 실행으로 LoginServlet 등 컴파일 |
| JDBC 드라이버 설치 | mysql-connector-j.jar를 Tomcat lib/ 폴더에 배치 |
| 동작 확인 | http://localhost:8080/CAN/campuslogin.jsp 접속 및 전체 기능 확인 |

## 5. 개발 표준

### 5.1 코딩 컨벤션

| 항목 | 규칙 |
|------|------|
| Java 클래스명 | PascalCase (예: `LoginServlet`, `DBUtil`) |
| 변수명 | camelCase (예: `loginUser`, `assetNo`) |
| JSP 파일명 | 소문자 + 언더스코어 (예: `main_student.jsp`, `room_reserve.jsp`) |
| SQL 파일명 | 기능 단위 명명 (예: `campusnav_db.sql`, `rooms_setup.sql`) |
| 주석 | 주요 로직에 한국어 주석 작성 |
| 인코딩 | UTF-8 (한글 처리 통일) |

### 5.2 DB 설계 표준

| 항목 | 규칙 |
|------|------|
| 테이블명 | 소문자 + 언더스코어 (예: `asset_transfer`, `room_reservations`) |
| 기본키 | AUTO_INCREMENT INT (예: `reserve_id`, `room_id`) |
| 문자셋 | utf8mb4 / utf8mb4_0900_ai_ci |
| 날짜 컬럼 | DATETIME 타입, DEFAULT CURRENT_TIMESTAMP |
| 외래키 | 참조 무결성 설정 (CONSTRAINT 명시) |
| 인덱스 | 자주 조회되는 컬럼에 KEY 설정 (asset_no, user_id, reserve_date 등) |

### 5.3 UI 설계 표준

| 항목 | 규칙 |
|------|------|
| 디자인 시스템 | ppd4 스타일 — 화이트 베이스 + 블루 포인트 |
| 기본 폰트 사이즈 | 15px |
| 공통 스타일 파일 | `/CAN/css/common.css` |
| UI 프레임워크 | Bootstrap 5.3.3 |
| 아이콘 | Bootstrap Icons |
| 색상 코드 | CSS 변수 (`--blue`, `--txt`, `--line2` 등) 통일 사용 |
| 반응형 | Bootstrap 그리드 시스템 적용, 모바일 대응 |

### 5.4 보안 표준

| 항목 | 규칙 |
|------|------|
| 인증 방식 | 서버 세션(HttpSession) 기반 인증 |
| 세션 유효시간 | 30분 (`setMaxInactiveInterval(30 * 60)`) |
| 역할 검증 | 모든 JSP 최상단 세션 체크 및 역할 미스매치 시 리다이렉트 |
| 쿠키 | 아이디 저장 쿠키 30일, Path = "/" 제한 |
| 비밀번호 | 현재 평문 저장 → 추후 BCrypt 해싱 적용 예정 |

## 8. 확장 개발 계획

현재 시스템 구현 이후 아래 기능을 추가 개발 대상으로 관리한다.

| 기능 | 내용 | 선행 조건 |
|------|------|-----------|
| Google Maps 연동 | detail.jsp, transfer.jsp 지도 API 적용 | Google Maps API 키 발급 |
| 실내 GPS 연동 | 제1·2공학관 GPS 설치 후 실내 위치 표시 | GPS 장비 설치 완료 |
| QR 코드 기능 | QR 스캔 → 자산 상세 페이지 자동 이동 | QR 라벨 인쇄 환경 구성 |
| 관리자 도착 알림 | 예약자 도착 시 WebSocket 실시간 알림 | 알림 서버 구성 |
| 비밀번호 해싱 | 평문 저장 → BCrypt 해싱으로 보안 강화 | LoginServlet 재컴파일 |
