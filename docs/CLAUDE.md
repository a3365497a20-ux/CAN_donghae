세부 네이밍 규칙은 §문서 관리 규칙 참조.

## 수행할 역할

분석자, 설계자, 구현자, 일정관리자, 문서관리자 역할을 병행한다.

**작업 흐름**: 요구사항 접수 → 현황 파악(코드/문서 직접 확인) → 설계 문서 업데이트 → 코드 구현 → 동작 테스트 → 완료 보고

**핵심 원칙**: 분석 → 설계 → 구현 → 테스트를 한 사이클로 완료. 중간에 멈추지 않음. 단, 매우 큰 작업이거나 여러 옵션이 있을 경우 방향 먼저 확인.

## 개발 진행 원칙

1. **방법론**: 프로젝트 상위 방법론은 `docs/개발방법론_테드소와이즈.md` (정보공학 기반, Samsung SDS Innovator / BDD / ADR 참고). 데이터 중심, 프로세스 중심 접근. 산출물 종류·단계는 이 문서의 §2 SDLC를 기준으로 한다.

2. **요구사항 명확화 우선**: 모호한 요청은 질문 및 대안 제시로 구체화. 추정하지 않음

3. **현황 파악 우선**: 설계/구현 전 관련 코드와 문서를 직접 확인

4. **문서 병행 업데이트**: 구현 진행 중 관련 설계 문서 동시 업데이트. 큰 작업 완료 시 작업 일지 작성

5. **점진적 적용**: 200줄 이상 코드는 파일 1개씩 구현 후 계속 진행. 500줄 초과 시 파일 분리 제안

6. **작업 완료 보고**: 완료 후 수정 파일 목록, 주요 변경 사항, 테스트 결과 간단히 요약

7. **작업 범위 확인**: 여러 파일에 걸친 대형 작업이나 여러 옵션이 있는 경우 방향 먼저 확인

## 코드 작성 규칙

1. **변경전 코드 분석**: 원본 파일 먼저 분석 후 영향 범위 한번에 처리

2. **파일 최상단 헤더 필수**: 작성날짜(KST), 파일위치, 코드제목/목적
   ```
   // 2026-04-02 - FormuLearning - 모듈 설명
   // 파일 위치: src/module.py
   ```

3. **쿼리문 빈 줄 제거**: 하나의 쿼리는 중간 빈 줄 없이. 여러 DML 쿼리는 빈줄 2개로 분리

4. **스크립트 작성 규칙**:
   - 파일 헤더: shebang, 인코딩, 날짜, 경로, 목적 주석 필수
     ```python
     #!/usr/bin/env python3
     # -*- coding: utf-8 -*-
     # 2026-04-02 - FormuLearning - 스크립트 설명
     # 파일 위치: ai-agent/scripts/스크립트명.py
     ```
   - **저장 위치**:
     - 상시 도구, 다수 프로젝트 공용: `/home/jychang/bin/스크립트명.py` (예: `record-request.py`)
     - 프로젝트 전용 상시 도구: `ai-agent/scripts/스크립트명.py` (날짜 접두어 없음)
     - 1회성 스크립트: `ai-agent/scripts/yymmdd-스크립트명.{py,sh}`
     - 작업에 부속된 근거 스크립트 (POC 등): 해당 작업 디렉토리 `ai-agent/claude/yymmdd_주제/script/`
   - DB 접속 정보는 `.env`에서 읽기

5. **커밋 메시지 형식**: Conventional Commits 사용
   - `feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:`

## 문서 관리 규칙

1. **문서 내용의 구체성**: 계획서는 파일명/클래스/함수명 수준까지 명시. 소스코드 포함 금지

2. **문서명 규칙**: `분류_yymmdd_제목 vX.X.md`
   - 예: `설계_260402_시스템 아키텍처 v1.0.md`
   - 업데이트: minor version 증가 (v1.1, v1.2...). 버전 올리자 요청 시: major version 증가

3. **문서 헤더 형식**:
   ```markdown
   # 문서 제목

   **파일 위치**: `docs/분류_날짜_제목.md`
   **작성일**: YYYY-MM-DD
   **버전**: v1.0
   **설명**: 한 문장 설명

   ---
   ```

4. **디렉토리별 용도**:

   **`docs/`** — **승인된 산출물 저장소**. 개발방법론 SDLC의 기획·분석·설계·운영 확정 문서만 배치. **Claude는 `docs/`에 직접 파일을 생성·수정하지 않는다.** 작성 중인 문서가 `docs/` 대상(개발방법론 §2의 공식 산출물)에 해당한다고 판단되면 사용자에게 먼저 확인한다. 승인된 경우에만 `ai-agent/claude/` → `docs/`로 이동한다.

   **`docs/old/`** — `docs/` 문서의 이전 버전 보관

   **`ai-agent/claude/`** — **Claude 작업 디렉토리**. 검토·작업계획·일지 등 작업 중인 문서를 여기에 작성.
   - `검토_yymmdd_주제.md`, `작업계획_yymmdd_주제.md`, `일지_yymmdd_주제.md`
   - 해당 문서가 `docs/` 공식 산출물로 확정될 만한 경우, 사용자에게 확인 후 승격

5. **디렉토리 명명 규칙**:

   - **참고자료 디렉토리** (외부 원본, 보관 산출물): **날짜 접두어 없음**, 단순 명쾌한 이름
     - 예: `ai-agent/uploads/`, `ai-agent/claude/260417_R_Python이식/src/pharmai_DoE_R_org/`
   - **작업 기록 디렉토리** (검토·일지·계획 + 근거 스크립트·산출물 묶음): **`yymmdd_주제/`** 형식
     - 예: `ai-agent/claude/260413_구시스템이관_디컴파일/`, `ai-agent/claude/260417_R_Python이식/`
     - 작업에 여러 자료가 얽히는 경우 하위에 `script/`, `src/`, `data/` 같은 단순 이름 서브 디렉토리 허용
   - **예외: 여러 날짜에 걸친 로그성 자료**: 주제명 단독 허용
     - 예: `ai-agent/claude/요청사항_로그/`

6. **요청사항 기록 (필수)**: → 자동 실행 명령어 섹션 참조

7. **네이밍 철학**: 한국어 사용자가 주 운영자이므로 파일명·디렉토리명에 **한글을 허용·권장**한다. 분류 접두어(`분석_`, `검토_`, `설계_`, `일지_` ...)로 정렬 시 주제별로 묶이도록 하고, 날짜는 `YYMMDD` 6자리로 파일명 길이를 절감한다. (상세 근거: `docs/개발방법론_테드소와이즈.md` §4.3)

8. **결정 기록 (ADR 성격)**: 기술 선택, 구조 변경, 이식·재설계 방침 등 **되돌아볼 가능성이 있는 결정**은 해당 작업 디렉토리의 검토·설계 문서에 `배경 / 대안 / 결정 근거` 형식으로 남긴다. 코드 주석·커밋 메시지로만 남기지 않는다. 공식 ADR 산출물은 개발방법론 §2.3 `Decision Records` 를 따른다.

9. **아이디어 제안**: 개선 필요 사항이나 대안이 있으면 검토 및 제안 가능

## 자동 실행 명령어 (승인 불필요)

### 요청사항 기록 (필수, 자동 실행)

사용자의 **모든 발언**을 즉시 기록. 사용자 발언 직후 **첫 번째 도구 호출**로 실행 (예외 없음).

**원칙 — 원문 그대로**: 사용자 발언은 **원문을 그대로** 기록한다. 요약·축약·생략·문장 다듬기·번역·의역 모두 금지. 길이에 상관없이 메시지 전체를 (줄바꿈·기호·반복·오탈자까지) 그대로 stdin으로 전달한다. 슬래시 명령(`/foo`), 첨부 마커(`@경로`), 한·영 혼용 모두 보존.

**방법 — HEREDOC 사용**: `record-request.py`는 stdin을 그대로 받아 적는다. `echo "..."` 한 줄 형태는 따옴표·줄바꿈 escape 문제와 요약 유혹 때문에 **사용하지 않는다**. 항상 HEREDOC으로 호출한다.

```bash
/home/jychang/bin/record-request.py <<'REQEND'
<사용자 발언 원문 그대로 — 줄바꿈·따옴표·기호 모두 보존>
REQEND
```

`'REQEND'` 처럼 종료 토큰을 작은따옴표로 감싸 변수 치환을 끄고, 본문에 `REQEND`가 등장할 가능성이 있으면 다른 토큰(예: `__END__`)으로 바꿔 사용한다.

파일: `ai-agent/claude/요청사항_로그/요청사항_yymmdd.txt` (세션당 하나)

### 읽기 전용 명령

- DB SELECT 쿼리 (데이터 조회만)
- `docker compose logs`, `docker ps`, `docker stats`
- 컨테이너 내 읽기 전용 명령 (grep, cat, ls, find 등)
- `date`, `git status`, `git log`, `git diff`
- `gh issue list`, `gh issue view`, `gh api repos/...`

### 자동 실행이지만 기록성 쓰기 동작 (승인 불필요)

- **gh issue edit** (본문 수정), **gh issue comment** (코멘트 추가) — 이슈 자체의 생성·상태 변경이 아닌 내용 갱신·보고 목적으로 자유 수행

### 승인 필요한 명령어

- INSERT, UPDATE, DELETE 등 DB 변경
- 스키마 변경 (ALTER, CREATE, DROP) → 반드시 마이그레이션 시스템 사용
- 컨테이너 재시작/중지 (restart, stop, start)
- 파일 수정/삭제 (Edit, Write, rm 등)
- 코드 배포
- **gh issue create / gh issue close** (사용자 승인 후 실행)
- **git commit/push** (사용자 승인 후 실행)
