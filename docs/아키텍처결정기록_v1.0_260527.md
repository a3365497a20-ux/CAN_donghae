## ADR-001 백엔드 기술 스택: JSP + Java Servlet 채택

| 항목 | 내용 |
|------|------|
| **결정일** | 2026-03 |
| **상태** | 확정 (Accepted) |
| **결정자** | 팀 전체 합의 |

### 배경 (Context)
교내 자원 내비게이션 시스템의 백엔드 기술 스택을 선택해야 한다. 팀원 전원이 Java 기반 웹 개발 교육 과정을 이수하였으며, 학교 실습 환경이 Apache Tomcat으로 구성되어 있다.

### 검토한 대안 (Alternatives)

| 대안 | 장점 | 단점 |
|------|------|------|
| JSP + Java Servlet | 팀 숙련도 높음, 학교 환경 호환 | 최신 프레임워크 대비 코드량 많음 |
| Spring Boot | 생산성 높음, 생태계 풍부 | 팀 학습 곡선, 환경 설정 복잡 |
| Node.js (Express) | 경량, JSON 친화 | 팀 숙련도 낮음 |
| Python (Django/Flask) | 간결한 코드 | 학교 WAS 환경과 불일치 |

### 결정 및 근거 (Decision & Rationale)
**JSP + Java Servlet 채택.** 팀 전원의 기술 숙련도와 학교 Tomcat 환경에 최적화된 선택이다. 별도 프레임워크 학습 없이 즉시 개발 착수 가능하며, 수동 컴파일 방식(compile.bat)으로 빌드 도구 없이 동작한다.

### 제약 사항 (Constraints)
- Servlet 수정 시 매번 수동 재컴파일 + Tomcat 재시작 필요
- 복잡한 의존성 주입, AOP 등 고급 기능 미지원
- JSP는 View/Controller 혼합 가능성 — 명확한 역할 분리 필요

## ADR-002 데이터베이스: MySQL 8.x + JDBC 직접 연결 채택

| 항목 | 내용 |
|------|------|
| **결정일** | 2026-03 |
| **상태** | 확정 (Accepted) |
| **결정자** | 팀 전체 합의 |

### 배경 (Context)
자산 8,401건, 이관 140건, 폐기 2,721건 등 대용량 데이터를 관리할 DBMS와 연결 방식을 결정해야 한다.

### 검토한 대안 (Alternatives)

| 대안 | 장점 | 단점 |
|------|------|------|
| MySQL + JDBC 직접 | 교육 환경 친화, 명시적 SQL 제어 | 연결 관리 코드 반복, Connection Pool 미사용 |
| MySQL + MyBatis | SQL 분리, 유지보수 용이 | 추가 설정 및 학습 필요 |
| MySQL + JPA/Hibernate | ORM으로 생산성 향상 | 학습 곡선 높음, 설정 복잡 |
| PostgreSQL | 고급 기능 풍부 | 학교 환경에 MySQL이 표준 |
| H2 (In-memory) | 설치 불필요 | 영속성 없음, 실 운영 부적합 |

### 결정 및 근거 (Decision & Rationale)
**MySQL 8.x + JDBC 직접 연결 채택.** DBUtil.java로 연결 공통화하여 반복 코드를 최소화한다. PreparedStatement 파라미터 바인딩으로 SQL Injection을 방지한다.

### 제약 사항 (Constraints)
- Connection Pool 미사용으로 동시 접속자 증가 시 성능 저하 가능
- ORM 미사용으로 SQL을 직접 관리해야 함

## ADR-003 프론트엔드: Bootstrap 5.3.3 + ppd4 디자인 시스템 채택

| 항목 | 내용 |
|------|------|
| **결정일** | 2026-03 |
| **상태** | 확정 (Accepted) |
| **결정자** | 팀 전체 합의 |

### 배경 (Context)
6개 역할, 19개 페이지에 걸쳐 일관된 UI를 빠르게 구축해야 한다. 별도 프론트엔드 빌드 시스템 없이 CDN으로 즉시 사용 가능한 환경이 필요하다.

### 검토한 대안 (Alternatives)

| 대안 | 장점 | 단점 |
|------|------|------|
| Bootstrap 5 (CDN) | 반응형 내장, 컴포넌트 풍부, CDN 즉시 사용 | 커스터마이징 다소 제한 |
| Tailwind CSS | 유틸리티 퍼스트, 자유도 높음 | 빌드 도구 필요, 학습 필요 |
| 직접 CSS 작성 | 완전한 자유도 | 개발 시간 과다 |
| React / Vue | SPA, 동적 UI | 빌드 환경 복잡, JSP와 통합 어려움 |

### 결정 및 근거 (Decision & Rationale)
**Bootstrap 5.3.3 CDN + ppd4 커스텀 디자인 시스템 채택.** CDN으로 빌드 도구 없이 즉시 적용 가능하며, `css/common.css`에 ppd4 변수(화이트/블루 포인트, CSS 변수 통일)를 정의하여 전 페이지 일관성을 확보한다.

### 제약 사항 (Constraints)
- CDN 의존으로 오프라인 환경에서 스타일 미적용 가능
- Bootstrap 기본 스타일 오버라이드 필요 시 specificity 충돌 주의

## ADR-004 길찾기 구현: HTML5 Canvas 기반 평면도 오버레이 채택

| 항목 | 내용 |
|------|------|
| **결정일** | 2026-03 |
| **상태** | 확정 (Accepted) |
| **결정자** | 팀 전체 합의 |

### 배경 (Context)
캠퍼스 평면도 위에 출입구 마커와 경로를 시각화해야 한다. Google Maps API는 키 미발급 상태이며, 실내 GPS 하드웨어도 미설치 상태다.

### 검토한 대안 (Alternatives)

| 대안 | 장점 | 단점 |
|------|------|------|
| HTML5 Canvas 오버레이 | 외부 의존 없음, 커스텀 자유도 최대 | 좌표 수작업 관리 필요 |
| Google Maps API | 실제 지도 기반, 검색 연동 | API 키 필요, 실내 지원 미흡 |
| SVG 기반 지도 | 벡터 확대 가능, 클릭 처리 용이 | 초기 SVG 제작 공수 큼 |
| Leaflet.js | 오픈소스 지도 라이브러리 | 커스텀 평면도 적용 복잡 |

### 결정 및 근거 (Decision & Rationale)
**HTML5 Canvas + 평면도 이미지(PNG) 오버레이 채택.** 외부 API 없이 즉시 구현 가능하며, 관리자가 직접 평면도를 클릭하여 경유점을 등록하는 방식으로 확장성을 확보한다. 경로 좌표는 JSON으로 DB에 저장하여 재사용한다.

### 제약 사항 (Constraints)
- 평면도 이미지 해상도와 Canvas 좌표 간 스케일 보정 필요 (모바일 대응)
- Google Maps 연동 시 추가 개발 필요

## ADR-005 세션 관리: HttpSession 서버 세션 채택

| 항목 | 내용 |
|------|------|
| **결정일** | 2026-03 |
| **상태** | 확정 (Accepted) |
| **결정자** | 팀 전체 합의 |

### 배경 (Context)
6개 역할의 사용자 인증 상태를 유지하고 역할별 접근을 제어해야 한다.

### 검토한 대안 (Alternatives)

| 대안 | 장점 | 단점 |
|------|------|------|
| HttpSession (서버 세션) | Java EE 표준, 구현 단순 | 서버 메모리 사용, 스케일아웃 불리 |
| JWT (토큰 기반) | Stateless, 확장성 좋음 | 구현 복잡, 토큰 무효화 어려움 |
| 쿠키만 사용 | 서버 저장 불필요 | 보안 취약, 조작 가능 |

### 결정 및 근거 (Decision & Rationale)
**HttpSession 채택.** 단일 서버 로컬 환경에서 가장 간단하고 안전한 방식이다. 세션 유효시간 30분을 설정하고, 모든 보호 페이지 최상단에서 세션 null 체크를 통해 미인증 접근을 차단한다.

### 제약 사항 (Constraints)
- 서버 재시작 시 세션 초기화 → 재로그인 필요
- 다중 서버 환경 확장 시 세션 공유 별도 처리 필요
