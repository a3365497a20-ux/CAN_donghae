### PR-01 사용자 인증 프로세스

**개요:** 사용자가 아이디·비밀번호·역할을 입력하여 시스템에 로그인하는 프로세스

**흐름도:**

```
[시작]
  │
  ▼
사용자가 아이디·비밀번호·역할 입력
  │
  ▼
[아이디·비밀번호 입력 여부 확인]
  │
  ├── 미입력 → "아이디와 비밀번호를 입력해 주세요" 오류 표시 → [시작]으로 돌아감
  │
  └── 입력됨 ▼
  │
  ▼
DB에서 user_id로 사용자 조회
  │
  ├── 없음 → "아이디 또는 비밀번호가 올바르지 않습니다" → [시작]
  │
  └── 있음 ▼
  │
  ▼
비밀번호 일치 확인
  │
  ├── 불일치 → "아이디 또는 비밀번호가 올바르지 않습니다" → [시작]
  │
  └── 일치 ▼
  │
  ▼
[역할 선택 여부 확인]
  │
  ├── 미선택 → "역할을 선택해 주세요" → [시작]
  │
  └── 선택됨 ▼
  │
  ▼
선택 역할 vs DB 역할 일치 확인
  │
  ├── 불일치 → "선택한 역할과 계정 유형이 일치하지 않습니다" → [시작]
  │
  └── 일치 ▼
  │
  ▼
기존 세션 무효화 → 새 세션 생성
세션 저장: loginUser, loginName, loginRole
세션 유효시간: 30분
  │
  ▼
[아이디 저장 선택 여부]
  │
  ├── 선택 → savedId 쿠키 30일 저장
  └── 미선택 → savedId 쿠키 삭제
  │
  ▼
역할에 따라 메인 페이지로 redirect
[종료]
```

**역할별 분기:**

| role | 이동 페이지 |
|------|------------|
| student | main_student.jsp |
| assistant | main_assistant.jsp |
| professor | main_professor.jsp |
| admin | main_admin.jsp |
| visitor | main_visitor.jsp |
| guest/기타 | main_guest.jsp |

### PR-03 자산 검색 프로세스

**개요:** 사용자가 키워드·분류·상태 조건으로 교내 자산을 검색하는 프로세스

**흐름도:**

```
[시작]
  │
  ▼
검색 조건 입력 (keyword, type, status)
  │
  ▼
SearchServlet에서 조건 파라미터 수신
  │
  ▼
[조건 조합]
  ├── keyword 있음 → item_name, asset_no, location_path LIKE 조건 추가
  ├── type 있음 → asset_type = 조건 추가
  └── status 있음 → asset_status = 조건 추가
  │
  ▼
assets 테이블 SELECT (WHERE 조건 + LIMIT 20 OFFSET (page-1)*20)
  │
  ▼
[검색 결과 여부]
  │
  ├── 0건 → "검색 결과가 없습니다" 메시지 표시
  │
  └── 결과 있음 ▼
  │
  ▼
search.jsp 렌더링
(자산 목록 테이블, 전체 건수, 현재 페이지, 페이지네이션)
  │
  ▼
[행 클릭]
  │
  └── detail.jsp?id={asset_no} 이동
[종료]
```

**페이지네이션 로직:**

```
totalCount = SELECT COUNT(*) 결과
totalPages = CEIL(totalCount / 20)
currentPage = request.getParameter("page") (기본값 1)
offset = (currentPage - 1) * 20
```

### PR-04 자산 예약 프로세스

**개요:** 사용자가 특정 자산에 대해 날짜·시간을 선택하여 예약을 신청하는 프로세스

**흐름도:**

```
[시작]
  │
  ▼
[세션 확인]
  │
  ├── 세션 없음 → 로그인 페이지로 redirect
  │
  └── 세션 있음 ▼
  │
  ▼
자산 상세 페이지(detail.jsp)에서 [예약하기] 클릭
  → reserve.jsp?id={asset_no} 이동
  │
  ▼
날짜·시작시간·종료시간 선택
  │
  ▼
[실시간 중복 체크 — AJAX]
  ├── 요청: GET /reserve?check=1&asset_no=...&reserve_date=...&start_time=...&end_time=...
  ├── DB 조회: reservations에서 동일 자산 + 날짜 + 시간대 겹침 확인
  │     SQL: SELECT COUNT(*) WHERE asset_no=? AND reserve_date=?
  │           AND NOT (end_time <= ? OR start_time >= ?)
  │
  ├── 중복(conflict=true)
  │     → 빨간 배경 + "해당 시간대에 이미 예약이 있습니다" 표시
  │     → 예약 신청 버튼 비활성화
  │
  └── 가능(conflict=false)
        → 초록 배경 + "예약 가능합니다" 표시
        → 예약 신청 버튼 활성화
  │
  ▼
[예약 신청 버튼 클릭]
  │
  ▼
POST /reserve (asset_no, reserve_date, start_time, end_time, purpose, phone)
  │
  ▼
ReserveServlet: 최종 중복 체크 재확인 (동시 접근 방지)
  │
  ├── 중복 발생 → 오류 메시지 반환
  │
  └── 이상 없음 ▼
  │
  ▼
reservations 테이블 INSERT
status = '예약완료'
  │
  ▼
예약 완료 메시지 표시
[종료]
```

## 3. 세션 및 접근 제어 프로세스

### 3.1 세션 체크 공통 프로세스

모든 보호 페이지(로그인 필요 화면) 최상단에서 아래 프로세스를 공통 수행한다.

```
[페이지 접근]
  │
  ▼
session.getAttribute("loginUser") 조회
  │
  ├── NULL → 세션 없음 → campuslogin.jsp로 redirect → [종료]
  │
  └── 값 있음 ▼
  │
  ▼
session.getAttribute("loginRole") 조회
  │
  ├── 해당 페이지 접근 불가 역할 → 홈 또는 접근 거부 페이지로 redirect
  │
  └── 접근 가능 역할 → 페이지 정상 렌더링
[계속 진행]
```
